## Version 7: Full Context (Collaboration Focus)

Hey folks üëã

Been working on something for local LLM users - wanted to share and get your ideas.

**Bodhi App** - like Ollama, but with OAuth 2.1 security baked in from day one (powered by llama.cpp). This means any web app on the internet can securely access your local model's inference, embeddings, and tools APIs - with your explicit consent.

Zero cloud dependency. Zero API costs. Your model, your machine, your data.

Think of it as turning your local LLM into a personal API server that third-party apps can talk to - but only with your permission. You control which apps get access, and you can revoke it anytime.

**Bodhi Chat** - a PoC to show devs what the Bodhi platform can do. Static site on GitHub Pages, connects to your local Bodhi App for deep research, agentic chat with web search. No backend needed from the app developer's side.

The core idea: decouple AI from the app. Users bring their own model and compute. Developers build the experience. Check out the **bodhi-js SDK** to build your own apps on this pattern.

## Version 10: Architecture Focus (For technical groups)

Built a demo of client-controlled AI architecture üëá

**Stack:**
- Bodhi App: Local LLM server (llama.cpp + OAuth 2.1). Exposes inference/embeddings/tools APIs under OAuth 2.1 security.
  GitHub: https://github.com/BodhiSearch/BodhiApp
- Bodhi Chat: Static React site (GitHub Pages) with OAuth 2.1 PKCE auth. Deep research + web search.
  GitHub: https://github.com/bodhiapps/chat
- bodhi-js: React SDK wrapping OAuth flow + OpenAI client
  GitHub: https://github.com/BodhiSearch/bodhi-js

**Flow:**
1. User installs Bodhi App locally, loads model
2. Web app redirects to OAuth (hosted identity)
3. User approves scope, web app gets token
4. Web app calls user's local server with OpenAI-compatible requests

**Result:** User consent-based access. User controls compute, model, data. Developer builds UX.

# Review: services crate auth-scope layer (HEAD~2..HEAD)

**Scope**: `crates/services/src/app_service/` + `crates/services/src/auth/auth_context.rs`
**Commits reviewed**: `d3279cf84..6d559f7b9`

## Findings

| Priority | File | Location | Issue | Recommendation |
|----------|------|----------|-------|----------------|
| P1 | `crates/services/src/app_service/auth_scoped_tokens.rs` | `create_token`, `list_tokens`, `get_token`, `update_token` | `AuthScopedTokenService` bypasses `TokenService` and calls `db_service()` directly for list/get/update. `create_token` (lines 49-92) inlines token generation logic (random bytes, SHA-256 hashing, prefix extraction, ULID generation) that belongs in `TokenService`. This breaks the service abstraction: `TokenService` trait exists with matching methods (`create_api_token`, `list_api_tokens`, `get_api_token_by_id`, `update_api_token`) but is never used. The passthrough accessor `token_service()` on `AuthScopedAppService` (line 117-119) exposes raw `TokenService` which is inconsistent with the scoped pattern used by mcps/tools/users. | Move token generation logic into `TokenService::create_token(user_id, name, scope)` so it owns the business logic. Change `AuthScopedTokenService` to delegate to `app_service.token_service()` instead of `app_service.db_service()` directly. |
| P2 | `crates/services/src/app_service/auth_scoped_users.rs` | `clear_sessions_for_user` (lines 91-101) | Missing auth gate. All other methods in `AuthScopedUserService` call either `require_token()` or `require_user_id()` before performing operations. `clear_sessions_for_user` performs a privileged admin action (clearing another user's sessions) without any authentication check. An `Anonymous` auth context would pass through unchecked. | Add `self.require_token()?;` or `self.require_user_id()?;` at the start of the method, consistent with the other methods in the service. |
| P2 | `crates/services/src/app_service/auth_scoped.rs` | passthrough accessors (lines 60-128) | `AuthScopedAppService` exposes raw `mcp_service()`, `tool_service()`, and `token_service()` as passthrough accessors alongside the auth-scoped `mcps()`, `tools()`, and `tokens()` accessors. This allows route handlers to accidentally bypass auth scoping by calling `auth_scope.mcp_service().list(user_id)` instead of `auth_scope.mcps().list()`. | Consider removing the passthrough accessors for services that have auth-scoped counterparts (`mcp_service()`, `tool_service()`, `token_service()`), or document the intended usage boundary clearly. The `auth_service()` passthrough may also be redundant given `users()`. |
| P3 | `crates/services/src/app_service/auth_scoped.rs` | `tokens()`, `mcps()`, `tools()`, `users()` (lines 44-58) | Each call to a sub-service accessor clones both `Arc<dyn AppService>` and `AuthContext`. If a handler calls `.tokens()` multiple times, each call creates a new `AuthScopedTokenService` with redundant clones. While Arc clones are cheap, `AuthContext` clone includes multiple `String` fields (user_id, client_id, token, username, etc.). | Consider caching sub-services or documenting that callers should bind the result to a local variable. Low impact but worth noting for hot paths. |

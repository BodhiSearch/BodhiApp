# MCP-Compliant Stateful Proxy Endpoint Plan

## Context

BodhiApp manages user-configured MCP server connections. Currently, tool execution uses a custom REST endpoint (`/{id}/tools/{tool_name}/execute`) with one-off connections. We need `/bodhi/v1/mcps/{id}/mcp` — a full MCP Streamable HTTP endpoint with stateful sessions, bidirectional communication, and server push support.

## Architecture Overview

The proxy creates a **bidirectional bridge** between a downstream MCP client and an upstream MCP server:

```
Downstream Client (Claude Desktop, Cursor, etc.)
    ↕ MCP Streamable HTTP (POST/GET/DELETE + SSE)
StreamableHttpService + LocalSessionManager
    ↕ ServerHandler methods
McpProxyHandler (implements rmcp::ServerHandler)
    ↕ Upstream MCP client (rmcp client connection)
Upstream MCP Server (configured by user)
```

Each session:
1. Client POSTs `initialize` to `/bodhi/v1/mcps/{id}/mcp`
2. Proxy creates session, connects to upstream MCP server as client
3. All subsequent requests forwarded bidirectionally
4. Upstream server notifications pushed to downstream via SSE
5. Session closed on DELETE or timeout

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Protocol | Full stateful MCP Streamable HTTP | POST + GET (SSE push) + DELETE, Mcp-Session-Id, resumability |
| Implementation | Use rmcp `StreamableHttpService` | Handles all protocol details: sessions, SSE streams, event IDs, resumability, priming events. Reimplementing would be error-prone. |
| Session manager | `LocalSessionManager` (in-memory) | Works for single-instance. Cluster needs sticky sessions by `Mcp-Session-Id` header or distributed SessionManager (tech debt). |
| Handler | `McpProxyHandler` implementing `ServerHandler` in `routes_app` | Needs access to both rmcp server types and AppService |
| Upstream per session | One persistent upstream connection per session | Established during `initialize`, lives for session duration |
| Notification bridge | Upstream `ClientHandler` holds downstream `Peer` | Upstream notifications forwarded to downstream client's SSE stream |
| Auth | Existing middleware (AuthScope) | Auth middleware puts `AuthContext` in request extensions; handler reads from HTTP Parts |
| Tools filter | Respected | `list_tools` filters, `call_tool` validates against filter |
| Instance mode config | Not added now | Deferred to tech debt |
| rmcp dependency | Add to `routes_app` with server features | `routes_app` needs both rmcp server types and AppService |

### Why rmcp StreamableHttpService (Not Custom Handler)

The stateful MCP protocol involves:
- Session ID generation and validation across POST/GET/DELETE
- Multiple simultaneous SSE streams per session with message routing
- Per-stream event ID assignment (hierarchical: `<index>/<http_request_id>`)
- Event caching in ring buffers for stream resumption via `Last-Event-Id`
- Priming events for client reconnection configuration
- Request-wise channels (per POST request) vs common channel (for GET stream / notifications)
- Message routing: responses/progress to request-wise channels, server notifications to common channel

rmcp's `StreamableHttpService` + `LocalSessionManager` implement all of this correctly. Reimplementing would be ~1000+ lines of tricky concurrent stream management code.

### Handler Context Injection

The `service_factory: Fn() -> Result<S, io::Error>` creates a handler per session with no request context. This is solved by **lazy initialization**:

1. Factory creates `McpProxyHandler` with `Arc<dyn AppService>` (shared) and `tokio::sync::OnceCell<UpstreamBridge>` (empty)
2. `initialize()` is always the first method called — reads `{id}` from URI in HTTP Parts, resolves upstream config, connects
3. All subsequent methods read from the initialized `OnceCell`

## Tech Debt Documentation

**File**: `ai-docs/claude-plans/202603/20260329-mcp-endpoint-techdebt.md`

#### `BODHI_INSTANCE_MODE=single|cluster`
- Add setting to distinguish single-process from multi-instance deployment.
- Current stateful MCP proxy uses `LocalSessionManager` (in-memory sessions per process).
- For cluster: requires sticky sessions (LB affinity by `Mcp-Session-Id` header) OR distributed SessionManager.
- If sticky sessions are guaranteed (e.g., K8s ingress session affinity), LocalSessionManager works as-is.
- Otherwise, implement Redis-backed `SessionManager` (rmcp's `SessionManager` trait is pluggable).

#### Cluster-specific constraints to solve
- **Session affinity**: MCP sessions are stateful. If client's requests hit different instances, sessions won't be found (404). Need LB sticky sessions or distributed session state.
- **Upstream connection per instance**: Each session has a persistent upstream connection on the instance that created it. Cannot be migrated between instances without reconnecting.
- **Notification routing**: Upstream notifications are delivered to the instance holding the upstream connection. If client reconnects to a different instance, notifications are lost.
- **Event ID caching**: `LocalSessionManager` caches event IDs in memory for stream resumption. In cluster, cache is per-instance. Resumption only works on the same instance.
- **Session cleanup**: If an instance crashes, its sessions are lost. Clients must re-initialize. For graceful shutdown, sessions should drain before termination.

#### Connection pooling
- Pool persistent upstream MCP connections per server. Reuse across sessions.
- Handle connection health checks, eviction, and per-MCP-instance auth isolation.

#### CacheService for cluster
- `MokaCacheService` is in-memory per-process. For cluster: Redis/PostgreSQL-backed CacheService.
- Make CacheService trait pluggable with configurable backend based on `BODHI_INSTANCE_MODE`.

#### Deprecate legacy tool endpoints
- `POST /{id}/tools/{tool_name}/execute` → replaced by MCP `tools/call`
- `GET /{id}/tools` → replaced by MCP `tools/list`
- `POST /{id}/tools/refresh` → replaced by MCP `initialize`
- Same for `/apps/` variants.
- Frontend should migrate to using MCP protocol endpoint.

#### Resource/prompt access control
- Add allow/disallow toggles per MCP instance for resources, prompts, completions.
- Boolean flags controlling whether the proxy forwards these operation categories.

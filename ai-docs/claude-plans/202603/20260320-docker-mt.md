# Plan: Dedicated Multi-Tenant CPU-Only Docker Release Workflow

## Context

Currently, releasing the multi-tenant Docker image requires triggering the full Docker release pipeline (`docker-dev/v*` or `docker/v*` tags), which builds ALL variants (cuda, rocm, vulkan, musa, intel, cann, cpu, multi-tenant). This is slow and wasteful when only the multi-tenant image needs updating. This plan adds a dedicated fast path for MT-only releases via a separate git tag namespace and workflow.

## Decisions

- **Coexistence**: Multi-tenant stays in existing full-release workflows. The new workflow is an independent fast path.
- **AMD64 only**: Multi-tenant targets cloud/SaaS (no ARM64).
- **Shared version counter**: MT releases share the version sequence with full Docker releases (via GHCR tag detection).
- **Both prod and dev**: `docker-mt/vX.Y.Z` (production) and `docker-mt-dev/vX.Y.Z` (development).
- **No GitHub Release**: Image push only — no release.json, no GH Release page entry.
- **Makefile targets**: `release.docker.mt-cpu` (prod), `release.docker.dev-mt-cpu` (dev).

## Changes

### 1. Modify `scripts/get_ghcr_version.py` (lines 43, 49)

Add `multi-tenant` to the regex variant groups so the shared version counter detects MT image tags.

**Line 43** (production regex):
```python
# Before:
r"^\d+\.\d+\.\d+-(cpu|cuda|rocm|vulkan)$"
# After:
r"^\d+\.\d+\.\d+-(cpu|cuda|rocm|vulkan|multi-tenant)$"
```

**Line 49** (development regex):
```python
# Before:
r"^\d+\.\d+\.\d+-(cpu|cuda|rocm|vulkan)-development$"
# After:
r"^\d+\.\d+\.\d+-(cpu|cuda|rocm|vulkan|multi-tenant)-development$"
```

Note: `tag.split("-")[0]` still correctly extracts `0.0.12` from `0.0.12-multi-tenant` (splits to `['0.0.12', 'multi', 'tenant']`).

### 2. Modify `Makefile.release.mk`

#### 2a. Parameterize `create_docker_release_tag` (lines ~293-298)

Add optional 3rd argument for tag base prefix (default: `docker`). Existing callers pass 2 args so default applies — no breakage.

```makefile
# Replace the hardcoded prefix logic with:
TAG_BASE="$(if $(3),$(3),docker)" && \
if [ "$(1)" = "production" ]; then \
    TAG_NAME="$$TAG_BASE/v$$NEXT_VERSION"; \
else \
    TAG_NAME="$$TAG_BASE-dev/v$$NEXT_VERSION"; \
fi && \
```

#### 2b. Add `.PHONY` entries

Add `release.docker.mt-cpu` and `release.docker.dev-mt-cpu` to the `.PHONY` declaration on line 4-5.

#### 2c. Add `release.docker.mt-cpu` target

Production MT-CPU release. Pattern mirrors `release.docker` but:
- Manual override var: `DOCKER_MT_VERSION` (format: `docker-mt/vX.Y.Z`)
- Calls `$(call create_docker_release_tag,production,$$CURRENT_VERSION,docker-mt)`
- Same `get_ghcr_docker_version,production` for shared version lookup

#### 2d. Add `release.docker.dev-mt-cpu` target

Development MT-CPU release. Pattern mirrors `release.docker-dev` but:
- Manual override var: `DOCKER_MT_DEV_VERSION` (format: `docker-mt-dev/vX.Y.Z`)
- Calls `$(call create_docker_release_tag,development,$$CURRENT_VERSION,docker-mt)`
- Same `get_ghcr_docker_version,development` for shared version lookup

### 3. Create `.github/workflows/publish-docker-mt-cpu.yml`

New workflow with 3 jobs (no matrix, no release creation). Modeled on `publish-docker.yml` but stripped down.

**Triggers:**
```yaml
on:
  push:
    tags:
      - 'docker-mt/v*'
      - 'docker-mt-dev/v*'
  workflow_dispatch:
    inputs:
      tag:
        description: 'Docker MT tag (e.g., docker-mt-dev/v0.0.1)'
        required: true
        type: string
```

**Concurrency:** `publish-docker-mt-cpu-${{ github.ref }}`

#### Job 1: `extract-version`
- Validate tag with pattern `'^docker-mt(-dev)?/v[0-9]+\.[0-9]+\.[0-9]+$'`
- Extract version: `docker-mt/v*` → production, `docker-mt-dev/v*` → development
- Outputs: `tag_version`, `build_variant`, `image_suffix`, `is_development`, `image_name`, `app_version`, `commit_sha`, `release_tag`

#### Job 2: `build-app-binary`
- Runner: `ubuntu-latest-8-cores`, timeout: 120min
- Dockerfile: `./devops/app-binary.Dockerfile`, platform: `linux/amd64`
- Tags: `app-binary-mt-<version><suffix>` and `app-binary-mt-latest<suffix>` (scoped to avoid collision with full pipeline)
- Cache scopes: `app-binary-mt-<build_variant>` and `app-binary-mt-deps`
- Build args: `BUILD_VARIANT`, `BODHI_VERSION`, `BODHI_COMMIT_SHA`, `CI_DOCKER=true`
- Output: `app_binary_tag`

#### Job 3: `build-multi-tenant`
- Runner: `ubuntu-latest-4-cores`, timeout: 90min
- Dockerfile: `./devops/multi-tenant.Dockerfile`, platform: `linux/amd64`
- Image tags:
  - `<version>-multi-tenant<suffix>` (e.g., `0.0.6-multi-tenant-development`)
  - `latest-multi-tenant<suffix>` (e.g., `latest-multi-tenant-development`)
- Cache scopes: `multi-tenant-mt-<build_variant>` and `multi-tenant-mt-deps`
- Build args: `BUILD_VARIANT`, `BODHI_VERSION`, `BODHI_COMMIT_SHA`, `APP_BINARY_IMAGE`, `CI_DOCKER=true`
- Verification step: `docker manifest inspect` on the published tag
- Output: `::notice` with image details

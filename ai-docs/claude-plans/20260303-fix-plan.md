# Multi-Tenant Isolation — Implementation Plan

## Context

The RLS commit (9223adb5d) added `tenant_id` columns to all tables and PostgreSQL RLS policies on 14 tables, but the application layer was left incomplete. Services hardcode `String::new()` as tenant_id in 14+ create/update paths, 16+ repository query methods lack tenant_id WHERE clauses, RLS `SET LOCAL` is never called at runtime, and tests run without tenant isolation. This plan completes the application-layer tenant isolation, standalone-first (all existing tests must pass), with multi-tenant runtime deferred to TECHDEBT.md.

**Reference implementations**: `AuthScopedTokenService` + `TokenRepository` (tokens domain) and `AccessRepository` (users domain) — both correctly thread tenant_id end-to-end.

**Scope**: Standalone mode only. Multi-tenant middleware branching, cross-tenant integration tests, and Anonymous tenant resolution are deferred.

**Key decisions** (from warp-plan-ctx.md):
- All resources per-tenant except settings (global)
- Token format: `bodhiapp_<base64url_random>.<client_id>` — client_id extracted from token for tenant resolution
- DbService is the transaction boundary — mutating methods wrap in txn with `SET LOCAL` (Postgres) / plain txn (SQLite)
- Modify original migrations directly for composite unique indexes (no backwards compat needed)
- Keep `get_standalone_app()` name (standalone is a domain term)

## Execution Model

Each layer is implemented by a **dedicated sub-agent** launched sequentially. The main agent orchestrates the pipeline:

### Sub-Agent Protocol
1. **Main agent launches sub-agent** for Layer N with:
   - Full layer spec (from this plan)
   - Consolidated summary of all previous layers' work (files changed, patterns established, key decisions made)
   - Reference files list (CLAUDE.md, PACKAGE.md for relevant crates)

2. **Sub-agent executes**:
   - Implements all items in the layer
   - Runs gate checks:
     - `cargo check --workspace` (compilation)
     - `cargo test -p <current_crate>` (current crate tests)
     - `cargo test -p <upstream_crates>` (all upstream crates that were modified in this or previous layers)
   - If tests pass: creates a local commit with message `multi-tenant: Layer N - <layer description>`
   - If tests fail: fixes issues and re-runs until green, then commits
   - Returns summary: files changed, key implementation decisions, any deviations from plan, test results

3. **Main agent consolidates**:
   - Merges sub-agent summary into running consolidated context
   - Launches next sub-agent with accumulated context

## Layer 1: DB / Core Infrastructure

### 1a. Transaction wrapper with SET LOCAL
**File**: `crates/services/src/db/default_service.rs`
- Add `async fn with_tenant_txn<F, R>(&self, tenant_id: &str, f: F) -> Result<R, DbError>` to `DefaultDbService`
- Inside: `db.begin()` -> if `get_database_backend() == Postgres`: `txn.execute_unprepared("SET LOCAL app.current_tenant_id = '{tenant_id}'")` -> `f(&txn)` -> `txn.commit()`
- SQLite: plain transaction (no SET LOCAL)
- Pattern proven in `crates/services/src/db/test_rls.rs` lines 263-286
- DB backend detection pattern established in `reset_all_tables()` (lines 49-98)
- Document pattern in `crates/services/CLAUDE.md`

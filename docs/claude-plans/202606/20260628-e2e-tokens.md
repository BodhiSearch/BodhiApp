The tokens-review sprint (`4dea5ea9..HEAD`) landed deny-by-default model grants and a unified
inference/grant-enforcement layer (see `docs/claude-plans/202606/screen-v2/tokens-review/`:

## Phase 1 — fix the 3 test-side breakages (one at a time)

For each: rebuild dev-server once up front (`make build.dev-server`), then run only that spec
(`HEADLESS=true npx playwright test --config=playwright.config.mjs --reporter=list --project standalone -- tests-js/specs/<path>` `-g "<title>"`),
confirm red → apply fix → confirm green. Keep a running was→cause→fix list.

### Fix 1 — `mcps-auth-restrictions.spec.mjs` Phase 5: restricted MCP is now 404, not 403
- **Cause (intended, F12/F31):** per-entity approval check was removed from
  `access_request_middleware.rs`; authorization moved to `AccessPolicy` in the handler.
  `mcps_show` (`crates/routes_app/src/mcps/routes_mcps.rs:84-88`) now returns
  `EntityError::NotFound("MCP")` for a non-listable instance — "don't reveal existence to a scoped
  token". The old error `access_request_auth_error-entity_not_approved` no longer exists.
- **Edit** `tests-js/specs/mcps/mcps-auth-restrictions.spec.mjs` ~line 254-262: rename the step to
  "Verify restricted MCP denied (404)"; assert `getResponseStatus()` `=== 404` and
  `error.code === 'entity_error-not_found'` (errmeta convention, `ErrorType::NotFound` → 404; confirm
  the exact code string at run time and match it).

### Fix 2 — `mcps-crud.spec.mjs:93`: playground shows friendly title "Echo Tool", not "echo"
- **Cause (intended):** `ToolDetail.tsx` renders `toolFriendlyTitle(tool)` (`t.title || annotations.title || t.name`).
  The everything-server `echo` tool has `title: "Echo Tool"`; `get_auth_info` has no title (so the
  header-auth call site already passes). The rail item is still keyed on the programmatic name.
- **Edit** `tests-js/specs/mcps/mcps-crud.spec.mjs:93`: keep `selectPlaygroundTool('echo')`; change the
  assertion to `expectPlaygroundToolSelected('Echo Tool')`. (Selection-by-name still drives the click;
  only the displayed-title assertion changes.)

### Fix 3 — `mcps-header-auth.spec.mjs:156`: `getPlaygroundResultContent` was renamed
- **Cause:** Playground-V2 migration renamed the method to `getPlaygroundResultRaw()`
  (`tests-js/pages/McpsPage.mjs:637`, selector `mcp-playground-result-raw`). The spec is the only
  remaining caller of the dead name.
- **Edit** `tests-js/specs/mcps/mcps-header-auth.spec.mjs:156`:
  `getPlaygroundResultContent()` → `getPlaygroundResultRaw()` (return value identical: stringified raw
  result JSON the test then `JSON.parse`s).

**Phase-1 gate:** rerun the three specs green, then the originally-failing folder set

## Phase 2 — grant coverage (after Phase 1 specs are green)

Grow black-box coverage, reusing existing page objects and the external-app `RESTPage` token-driven
flow (`app.rest.sendRequest(...)` → `getResponseStatus()`/`getResponse()`). One logical chunk per

### 2a — Rewrite the SDK-compat negative step into API-token-permission coverage (the 4th failure)
- **Why:** the "OAuth-only route rejects bodhiapp_ token" premise is intentionally obsolete — the apps
  router now accepts `Some(TokenScope::User)` and a default API token carries `McpGrant::All`, so it is
  grant-allowed on `/apps/mcps/*`. Per the operator: do **not** patch the old assertion; replace it
  with thorough coverage of the new API-token permission dimensions.
- **Edit** `tests-js/specs/mcps/mcps-sdk-compat-everything.spec.mjs` (the Phase-5 negative step ~586):
  mint API tokens with explicit grants via the UI (`TokensPage.createTokenWithGrants`) and assert the
  four dimensions end-to-end through the proxy + REST listings:
  - **`mcps_access` All / Specific{granted} / Specific{empty=deny}:** SDK `buildMcpClient` against a
    granted MCP **connects**; against a non-granted MCP **throws** (proxy `ensure_mcp_connect` → 403);
    an empty-grant token throws on the same MCP.
  - **`mcps_list`:** `GET /v1/mcps` (via REST page) lists only granted instances when `mcps_list` off /
    Specific; lists all when list-all/All.
  - **`models_access` / `models_list`:** `GET /v1/models` reflects the toggle; `POST /v1/chat/completions`
    on a granted model 200s, non-granted 403s.
  Keep the existing positive app-token SDK journey intact; only the negative step is replaced.

### 2b — F20: positive paths in `specs/tokens/app-tokens-grants.spec.mjs`
Current spec covers only the no-model **deny** path. Add `test.step`s (same single-test, many-steps
style): in setup register one API model + one Everything MCP instance; owner approves **one specific
model + one specific MCP** (+ list toggles); then via `app.rest`:
- granted model `POST /v1/chat/completions` → 200; a non-granted model → 403.
- `GET /v1/mcps` lists only the granted MCP; non-granted MCP get → 403/404.
- `GET /v1/models` reflects the toggle; granted MCP reachable, non-granted denied.
- assert `/bodhi/v1/user` `access.models`/`access.mcps` reflect the specific grants.

### 2c — F21: owner-extra-MCP path in `pages/AccessRequestReviewPage.mjs`
`approveWithGrants` has `listModels`/`modelsSpecific`/`listMcps` but no MCP-specific arm. Add
`setMcpAccessSpecific()` (mirror `setModelAccessSpecific()` incl. the overlay-detach wait) and an
`mcpsSpecific` option to `approveWithGrants`; exercise it from the F20 spec where the owner grants an
extra MCP the app didn't itemize.

### 2d — F41: list-all + All case in `specs/tokens/api-tokens.spec.mjs`
Add a `test.step` creating a token with list-all + `All` model and MCP access (via
`createTokenWithGrants`), then assert the token sees the full `/v1/models` + `/v1/mcps` listings and
infers on any registered model.

### 2e — F42: move keyboard-nav out of the inline CSS reach in `api-tokens.spec.mjs`
The spec reaches into `.l-listrow.active` inline (~line 148). Add `TokensPage.selectRowByKeyboard(dir)`

asserts the new selection through the `?select=` param + `detailRail` testid — no raw CSS-class reach.
Replace the inline block with the method call.

- **Verify notes**: The finding accurately identifies an asymmetry in this diff: the listing-filter pattern was added to OAI but not to the Anthropic/Gemini listing handlers. ### F9: HTTP verb mismatch: approve endpoint uses PUT while deny and revoke use POST
- **Priority**: important  ·  **Verdict**: confirmed  ·  **Category**: api-consistency
- **File**: `crates/routes_app/src/apps/routes_apps.rs`
- **Location**: utoipa annotation ~line 227 (put); crates/routes_app/src/routes.rs ~line 354 (put(apps_approve_access_request))
- **Issue**: The three owner-facing state-transition actions on an access request use inconsistent HTTP verbs: approve is PUT (utoipa and Axum route binding), while deny and revoke are POST. All three are one-shot non-idempotent transitions (a second call returns 409 Conflict), so none satisfies the RFC definition of idempotency that would justify PUT.
- **Failure scenario**: A client infers the approve verb from the sibling deny/revoke POST endpoints and sends POST /bodhi/v1/access-requests/{id}/approve → 405 Method Not Allowed. Auto-generated SDKs driven by OpenAPI also diverge in method name between approve (PUT) and deny/revoke (POST).
- **Recommendation**: Change approve to POST to match deny and revoke: update the utoipa annotation from put to post; change put(apps_approve_access_request) to post(apps_approve_access_request) in routes.rs. All three state-transition action endpoints on the same resource family should use the same verb.
- **Rationale**: REST convention: POST is correct for non-idempotent action endpoints that change state. Consumers and auto-generated SDKs derive the verb from the OpenAPI spec, so the mismatch creates a correctness hazard for any client handling these actions uniformly.
- **Evidence**: routes_apps.rs line 227: `#[utoipa::path(put, ...)]` for approve; routes.rs line 354: `put(apps_approve_access_request)`. Deny uses `post` (routes_apps.rs line 376, routes.rs line 358). Revoke (new in this diff) uses `post` (routes_apps.rs line 440, routes.rs line 364). Auth test in test_access_request_auth.rs line 9 explicitly encodes `("PUT", "/bodhi/v1/access-requests/test-id/approve")`, confirming the PUT is intentional. The git diff shows revoke was newly added with POST, deepening the existing approve-PUT vs deny-POST inconsistency.
- **Verify notes**: The inconsistency is real and confirmed. The `approve` endpoint predates this diff (already used PUT), and the newly added `revoke` endpoint uses POST, making three sibling action endpoints: approve=PUT, deny=POST, revoke=POST. A client that infers the approve verb from the deny/revoke pattern would send POST to /approve and receive 405 Method Not Allowed. Priority "important" is correct — it is a real API correctness hazard for SDK consumers and for any client uniformly handling these action endpoints, but not critical since the approve endpoint does function correctly when called with PUT and the OpenAPI spec accurately reflects the divergence.
- **Sources**: api:shapes

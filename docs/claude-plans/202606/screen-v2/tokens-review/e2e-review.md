### F21: AccessRequestReviewPage.approveWithGrants() has no mcpsSpecific path — owner-extra MCP picker entirely unexercised in E2E
- **Priority**: important  ·  **Verdict**: plausible  ·  **Category**: test-coverage
- **File**: `crates/lib_bodhiserver/tests-js/pages/AccessRequestReviewPage.mjs`
- **Location**: approveWithGrants(), lines 70-76
- **Issue**: approveWithGrants accepts listModels/modelsSpecific/listMcps but has no mcpsSpecific parameter. The review page renders the owner-extra MCP picker when mcps_access=true (testIdPrefix='review-mcp-access': review-mcp-access-mode-specific, review-mcp-access-panel-*, review-mcp-access-panel-done). No E2E test ever touches these selectors. Without it, the mcps_extra field in ApprovedResourcesV1 is never tested E2E.
- **Failure scenario**: A bug in the consent screen where clicking review-mcp-access-mode-specific opens an empty panel, or where review-mcp-access-panel-done submits the wrong field name (e.g. a copy-paste error mapping mcps_extra to models), would be completely undetected. The spec calls approveWithGrants with mcps_access: true in the requested scope, but the picker defaults to All without any E2E exercising the Specific path.
- **Recommendation**: Add a setMcpAccessSpecific() method mirroring setModelAccessSpecific(): click review-mcp-access-mode-specific, wait for panel to open, select an MCP, click Done, wait for review-mcp-access-panel to reach state:'hidden'. Add an mcpsSpecific option to approveWithGrants and use it in a test that exercises the full owner-extra MCP grant path end-to-end.
- **Rationale**: The plan explicitly requires the MCP grant path. The page object gap is the direct reason no E2E test covers mcps_extra.
- **Sources**: bug:e2e

### F24: reducedMotion:'reduce' set too late — inside 'List behaviors' step, missing coverage for the preceding 'Scoped grants' step
- **Priority**: important  ·  **Verdict**: confirmed  ·  **Category**: correctness
- **File**: `crates/lib_bodhiserver/tests-js/specs/tokens/api-tokens.spec.mjs`
- **Location**: line 137 (inside 'List behaviors' step); 'Scoped grants' step starts at line 106
- **Issue**: page.emulateMedia({ reducedMotion: 'reduce' }) is first applied at line 137 inside 'List behaviors'. The preceding 'Scoped grants' step (lines 106-133) calls selectSpecificFromPanel twice in sequence and openTokenRail — all operations that open/close Sheet overlays or trigger view-transition animations. The model-access Sheet close animation is still running when selectSpecificFromPanel('mcp-access') attempts to click the Specific radio, and the rail view-transition is live during the chip assertions.
- **Failure scenario**: On a browser with CSS animations enabled (the default before emulateMedia is called): (1) the model-access panel's Sheet close animation intercepts the click when the MCP-access Specific radio is targeted — causing a misfire or timeout. (2) openTokenRail triggers a view-transition; if the rail detail pane is still cross-fading, expectModelGrantChip asserts on a partially-mounted rail and fails intermittently.
- **Recommendation**: Move page.emulateMedia({ reducedMotion: 'reduce' }) to the very top of the outer test body (before the first test.step), so it covers all steps including 'Scoped grants'. Also add a panel-close wait in selectSpecificFromPanel after clicking accessPanelDone: await this.page.locator(this.selectors.accessPanel(prefix)).waitFor({ state: 'hidden' }) mirroring AccessRequestReviewPage.setModelAccessSpecific().
- **Rationale**: The emulateMedia call is scoped too late. Both the Sheet-close interception risk and the rail view-transition race are live during 'Scoped grants' but are never suppressed for that step.
- **Evidence**: In api-tokens.spec.mjs, `page.emulateMedia({ reducedMotion: 'reduce' })` appears exactly once, at line 137, inside the 'List behaviors' step. The preceding 'Scoped grants' step (lines 106-133) is fully unprotected.

Two concrete gaps in TokensPage.mjs:
1. `selectSpecificFromPanel` (lines 127-143) clicks `accessPanelDone` at line 141 then returns immediately with no `waitFor({ state: 'hidden' })` on the panel. `createTokenWithGrants` calls it for `model-access` then `mcp-access` back-to-back (lines 163-164), so the model-access Sheet slide-out animation is potentially live when the mcp-access Specific radio is clicked.
2. `openTokenRail` (lines 176-181) only calls `waitForSelector(detailRail)` to confirm DOM presence; no view-transition settle wait. `expectModelGrantChip`/`expectMcpGrantChip` assert immediately after, while the rail cross-fade can still be in progress.

All other V2 page objects (`ModelsListPageV2.mjs:62`, `SettingsPage.mjs:30`, `AllUsersPage.mjs:35`, `ChatPage.mjs:74`, `LocalDiscoveryPage.mjs:52`) call `emulateMedia` in their `navigate()` method, covering all steps. `app-tokens-grants.spec.mjs` places it at line 35, before any step. Only `api-tokens.spec.mjs` defers it to the 4th step, leaving 'Scoped grants' unprotected.
- **Verify notes**: The finding is fully confirmed. Both described race conditions are structurally present: (1) no panel-close wait in selectSpecificFromPanel after clicking done, creating a Sheet-close animation race for the sequential model-access → mcp-access picker flow; and (2) no view-transition settle after openTokenRail, creating a chip assertion race. The fix is twofold: move emulateMedia to before the first test.step, and add `await this.page.locator(this.selectors.accessPanel(prefix)).waitFor({ state: 'hidden' })` after the done click in selectSpecificFromPanel. Priority "important" is correct — these cause intermittent CI flakiness, not systematic failures.
- **Sources**: bug:e2e, prior:e2e-review F3

### F25: selectSpecificFromPanel does not wait for the picker Sheet overlay to detach before the next interaction
- **Priority**: important  ·  **Verdict**: confirmed  ·  **Category**: correctness
- **File**: `crates/lib_bodhiserver/tests-js/pages/TokensPage.mjs`
- **Location**: selectSpecificFromPanel(), after accessPanelDone click
- **Issue**: AccessPickerPanel is a shadcn Sheet with a fixed inset-0 z-50 overlay that animates out on close. After clicking Done, selectSpecificFromPanel returns immediately; createTokenWithGrants then opens the second (MCP) picker or clicks the scope card or Generate while the closing overlay may still intercept clicks. The sibling AccessRequestReviewPage.setModelAccessSpecific() correctly waits for '...-panel' to reach state:'hidden'; the TokensPage version omits that wait.
- **Failure scenario**: After closing the model-access panel, the overlay is still fading out when the code tries to click the MCP-access Specific radio. The click is intercepted by the invisible overlay, causing a misfire or element-not-interactable error. This is especially likely without reducedMotion active (see F24).
- **Recommendation**: After clicking accessPanelDone(prefix), add: await this.page.locator(this.selectors.accessPanel(prefix)).waitFor({ state: 'hidden' }) before returning, mirroring setModelAccessSpecific in AccessRequestReviewPage.
- **Rationale**: Avoids overlay-intercept flakiness — the same class of Sheet-detach race the consent review page already guards against.
- **Evidence**: In `TokensPage.mjs` line 141-142, `selectSpecificFromPanel()` clicks the Done button and returns immediately with no subsequent `waitFor({ state: 'hidden' })` on the panel locator. In contrast, `AccessRequestReviewPage.mjs` line 60 (newly added in this diff) explicitly adds `await this.page.locator('[data-testid="review-model-access-panel"]').waitFor({ state: 'hidden' })` after clicking Done, with the explicit comment "Wait for the Sheet overlay to detach so it no longer intercepts clicks." `createTokenWithGrants()` at lines 163-164 calls `selectSpecificFromPanel('model-access')` then immediately `selectSpecificFromPanel('mcp-access')` when both flags are true, so the model-access Sheet can still be animating closed when the code tries to click `mcp-access-mode-specific`. Both files are new code in the `4dea5ea9..HEAD` diff — not pre-existing.
- **Verify notes**: The gap is a straightforward omission: the same Sheet-detach guard was correctly written in `AccessRequestReviewPage.setModelAccessSpecific()` (same diff) but not carried into `TokensPage.selectSpecificFromPanel()`. Fix: after line 141 (`accessPanelDone` click), add `await this.page.locator(this.selectors.accessPanel(prefix)).waitFor({ state: 'hidden' })` before the return. Priority remains "important" — this causes E2E test flakiness (not a data/security issue) but is a real, reproducible race that will trigger on any CI run without `reducedMotion: 'reduce'` active.
- **Sources**: prior:e2e-review F2

### F43: INTEG_TEST_USER_MANAGER env vars not validated in beforeAll — silent failure in standalone project
- **Priority**: nice-to-have  ·  **Verdict**: confirmed  ·  **Category**: correctness
- **File**: `crates/lib_bodhiserver/tests-js/specs/tokens/api-tokens.spec.mjs`
- **Location**: lines 259-262 (manager credentials inline); beforeAll validation at lines 21-25 does not cover them
- **Issue**: The multi-user test builds managerCredentials with process.env.INTEG_TEST_USER_MANAGER and INTEG_TEST_USER_MANAGER_ID without any guard. The describe-level beforeAll validates only INTEG_TEST_USERNAME / _PASSWORD / _USERNAME_ID (via getTestCredentials) and apiKey. If the manager env vars are absent, the test fails at LoginPage.performOAuthLogin() with an obscure OAuth error rather than a loud 'Missing INTEG_TEST_USER_MANAGER' message. Convention: 'Never test.skip() for missing env vars; throw in beforeAll so failures are loud.'
- **Failure scenario**: CI environment with INTEG_TEST_USERNAME set but INTEG_TEST_USER_MANAGER absent: beforeAll passes (admin creds valid), standalone test runs, managerCredentials.username is undefined, LoginPage fails at the username fill step with a Playwright 'value must be a string' internal error. Root cause is invisible from the error message.
- **Recommendation**: In beforeAll (or in a guard at the top of the multi-user test body before the try block), throw if INTEG_TEST_USER_MANAGER or INTEG_TEST_USER_MANAGER_ID is falsy: if (!process.env.INTEG_TEST_USER_MANAGER) throw new Error('Missing INTEG_TEST_USER_MANAGER for multi-user test')
- **Rationale**: Convention requires throwing in beforeAll for missing env vars so that CI surfaces the configuration gap with a useful message rather than a confusing downstream failure.
- **Evidence**: File is in the diff (4dea5ea9..HEAD). The `beforeAll` at lines 21-25 calls `getTestCredentials()` (which throws for missing `INTEG_TEST_USERNAME`/`_PASSWORD`/`_USERNAME_ID`) and `getAuthServerConfig()` (which throws for missing auth URL vars), but performs no validation of `INTEG_TEST_USER_MANAGER` or `INTEG_TEST_USER_MANAGER_ID`. At lines 259-263, `managerCredentials` is built inline from `process.env.INTEG_TEST_USER_MANAGER` (→ `username`) and `process.env.INTEG_TEST_USER_MANAGER_ID` (→ `userId`) with no null check. The `auth-server-client.mjs` file shows the established pattern — `getTestCredentials()`, `getRealmAdminCredentials()`, `getMultiTenantConfig()` all throw explicitly for missing vars — but no equivalent helper exists or is called for the manager user. If `INTEG_TEST_USER_MANAGER` is absent, `managerCredentials.username` is `undefined`, which is passed to `LoginPage.performOAuthLogin()` and will fail with an obscure Playwright error rather than a clear configuration message.
- **Verify notes**: The finding is accurate and the code gap is real. Both missing env vars (`INTEG_TEST_USER_MANAGER`, `INTEG_TEST_USER_MANAGER_ID`) are consumed inline without any guard in `beforeAll` or in the step body before the `try` block. The project convention (documented in memory as `feedback_no_skip_for_missing_env.md`) explicitly requires throwing in `beforeAll` for missing env vars. Priority stays at nice-to-have: this is a test DX issue only — no production code is affected, the test still fails (just with a confusing message), and the manager env vars are likely always set in the integration CI environment where this test runs.
- **Sources**: bug:e2e

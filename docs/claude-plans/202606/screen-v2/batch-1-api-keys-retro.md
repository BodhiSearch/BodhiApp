## Surprises / learnings (carry forward)
- **Route tests render the page component directly, not through `__root`** â€” so `useShellChrome`'s
  published slots (headerActions/rail) don't appear unless the test wraps in `ShellSlotsProvider` +
  a small slots-consumer harness. Established that harness pattern in the V2 RTL tests; reuse it for
  every later screen that publishes chrome.
- **`routeTree.gen.ts` is git-ignored** â€” regenerated by the Vite plugin at dev/build (and via
  `@tanstack/router-generator`'s `Generator` API for a one-off). New route files are committed; the
  tree is not. E2E picks up the new route through live Vite, no rebuild.
- **Filter tabs collide with status badges on text** â€” the access-requests list renders both a
  "Pending" tab and "Pending" status badges, so `getByText('Pending')` is ambiguous. Target row
  status via `request-status-{status}` testids instead.
- **Generic design class names are real collision risk** â€” `.page-header`, `.filter-tab`,
  `.empty-state`, `.btn-accent` etc. Scoping `api-keys.css` under `.api-keys-screen` is the pattern;
  apply it to any future screen CSS with un-prefixed class names.
- **No backend changes** this batch â†’ no `make test.backend` needed; the 2 pre-existing backend
  live-test failures remain unrelated (techdebt.md).

## Reusable patterns to carry to later batches

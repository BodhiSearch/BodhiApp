## 1. Goal

Migrate the production frontend (`crates/bodhi`, Vite + TanStack Router + TanStack Query v5 +
shadcn/ui) from the current **sticky top-header + dropdown nav** to the V2 design: a **left-sidebar
`AppShell`** with 5 sibling nav sections (Chat Â· Models Â· MCP Â· API Keys Â· Settings), each with
collapse-to-icon-rail, hover column-resize, an optional 3rd detail "rail", and mobile drawers.

The migration is overwhelmingly **presentation-layer**: domain hooks already cover every
`/bodhi/v1/*` endpoint these screens need. We keep the data layer; we replace chrome + per-screen UI.
**End state:** all 14 screens on AppShell V2, all per-screen flags removed, all old chrome/dead code
deleted, all tests (unit + e2e) green.

## 2. Approach

- **In-place, flag-gated, per screen.** `AppShell` becomes the root layout (Batch 0). Each screen
  renders **old or new based on a per-screen flag** at the **same route** â€” both versions coexist
  while a batch is in flight. The flag is retired and the old version deleted once the new screen is
  accepted (end of its batch).
- **One nav section per batch**, in **roadmap riskĂ—traffic order**:
  `Batch 0 Foundation â†’ API Keys â†’ Settings â†’ Models â†’ MCP â†’ Chat`.
- **Iterative & manually coordinated.** No frozen master plan. Each batch runs the loop in Â§6,
  hands over context via the working folder, and ends by writing the **kick-off prompt for the next
  batch** so the next session starts exploratory with full context.
- **Reference APIs are a per-batch prerequisite**, not a Batch-0 deliverable. A batch that needs an
  external `api.getbodhi.app` capability defines its interface + spec + MSW stub **as that batch's
  prerequisite** (see Â§4).

## 4. Reference API & blockers

- **What it is:** an external service `https://api.getbodhi.app/` serving data the backend doesn't
  own (first need: the **MCP Discover catalog** â€” publishers, verified badges, install counts, tool
  lists, stats). The frontend calls it **directly** (no backend proxy), sending the user's OAuth
  **id_token** as identity for per-user rate-limiting/analytics. Base URL is configurable
  (env-overridable for tests).
- **Per-batch prerequisite (not Batch 0):** when a batch's screens need a reference-API capability,
  that batch's prerequisite step:
  1. Defines the TS interface (`src/hooks/reference/types.ts`) from the prototype's data shape.
  2. Writes an **interface spec doc** in the working folder (e.g.
     `screen-v2/mcp-discover-reference-api.md`: endpoints, request/response schemas, id_token auth)
     so the API team builds it in **parallel** (it'll be available in dev/prod separately).
  3. Builds & tests the UI against an **MSW external-origin stub** (Â§5) so the batch is never
     blocked on the real API.
- **Blocker handling:** if a real reference API isn't available in an environment where it's needed
  (e.g. CI e2e), the batch plan must state how it's handled (stub origin, tagged/gated e2e, or
  defer that screen) â€” explicitly, never silently.

## 5. Cross-cutting: e2e migration context

- **Black-box only** (UI interactions; no `page.evaluate`/`page.context` fetch). Suite lives in
  `crates/lib_bodhiserver/tests-js`; run `make test.e2e` (dev-server + Vite HMR, **no ui-rebuild**).
- **testid discipline** (the linchpin of presentation-only migration): when markup is restructured,
  **preserve `data-testid` + ARIA roles verbatim** so page objects keep working. Conventions: shell
  nav `data-testid="shell-nav-{section}"` / `shell-sub-{subPage}`; page root
  `data-testid="{domain}-page"` + `data-pagestatus`.
- **Per batch, e2e is part of the batch** (not deferred): each batch updates its section's page
  objects (`tests-js/pages/*.mjs` â€” nav switches old top-nav â†’ shell nav; dialogâ†’page renames) and
  specs, and they must pass before commit.
- **RTL+MSW**: handlers/fixtures reused unchanged (contract identical). External-origin reference
  stubs use plain `http` (not `typedHttp`); `mockAppInfo` returns a mock `reference_api_endpoint`
  (e.g. `http://localhost:9000`) so the client points at the stub purely via the normal `/info`
  flow â€” no env hacks. `mockUserLoggedIn` carries `id_token`.

## 6. The per-batch loop (the core of this playbook)

Each batch (a nav section) runs this loop, coordinated manually with check-ins between steps:

1. **Explore** â€” read the working folder (prior batch retrospectives + this batch's kick-off
   prompt), inspect what's already migrated, and survey this section's screens (design source +
   prod routes + existing hooks/tests).
2. **Prerequisites** â€” ensure everything this batch depends on is in place **before** implementing:
   shared components/CSS this section needs, reference-API interface+spec+stub (if any, Â§4), backend
   changes + OpenAPI/ts-client regen (if any), per-screen flags wired.
3. **Plan â†’ refine** â€” write this batch's plan to the working folder
   (`screen-v2/batch-N-<section>-plan.md`): screens, AppShell props per screen
   (`section`/`subPage`/rail?/`contentClass`/`resizeKey`), reused hooks, new pieces, test list
   (RTL + which MSW handlers + which e2e specs), risks. Present it; refine with the user; get
   approval before coding.
4. **Implement** â€” migrate the section's screens behind their flags following the recipe in Â§7.
5. **Migrate e2e + tests** â€” update RTL tests (keep testids) and the section's page objects/specs.
6. **Ensure all pass** â€” section RTL green (`cd crates/bodhi && npm run test`), backend tests if
   Rust changed (`make test.backend`), `make build.ts-client` if regen, e2e green
   (`make test.e2e`). Run **all** gate checks â€” never skip e2e before commit.
7. **Retire flags + commit** â€” remove this section's per-screen flags and delete its old
   versions/dead code, then commit per batch (trunk-based, directly on `main`; rebase first).
8. **Retrospect** â€” write `screen-v2/batch-N-<section>-retro.md`: what landed, divergences from the
   design (folded vs left local), surprises, reusable patterns, anything to fix centrally.
9. **Kick-off next batch** â€” write `screen-v2/batch-(N+1)-kickoff.md`: a prompt that starts the
   next batch exploratory, carrying forward learnings + prerequisites, so the next session re-enters
   this loop at step 1.

**Working-folder artifacts per batch:** `batch-N-<section>-plan.md`, `batch-N-<section>-retro.md`,
`batch-(N+1)-kickoff.md`, plus any reference-API interface spec the batch produced.

## 7. Per-screen migration recipe (used inside step 4)

Presentation-only â€” hooks, queries, mutations, fixtures unchanged.
1. Keep all hooks/queries and the `AppInitializer` wrapper (stays outside `<Shell>`).
2. Render via the per-screen flag: old version unchanged, **or** `<Shell section subPage resizeKey
   breadcrumb headerActions sidebar rail>` + ported design markup/CSS. Pick `contentClass`
   (`flush`+`mainScroll={false}` for full-height forms/lists/chat; default scroll otherwise).
3. **Dialog â†’ page** where the design dictates (New App Token â†’ `routes/tokens/new/`,
   `createFileRoute('/tokens/new/')`, real `useCreateToken` replacing the prototype's fake generator).
4. Port shared design components as TSX (`ModelAccessPicker.tsx` from `model-access-picker.jsx`).
5. **Strip on every port:** `lucide.createIcons()`, `Object.assign(window,â€¦)`,
   `ReactDOM.createRoot`, `data-theme` setattr effects, `TweaksPanel`/`useTweaks`/`EDITMODE`.
6. Preserve `data-testid` + ARIA verbatim across the restructure (Â§5).

## 9. Risks / watch-outs (carry into every batch)

1. **Never transition `grid-template-columns`** with `var()`/`minmax` tracks (Chromium sticks old
   width) â€” shell avoids it; keep the inline `setProperty` resize.
2. **Global `lucide`/`window` â†’ ESM**: grep each ported file for
   `lucide`/`window.`/`ReactDOM.createRoot`/`data-lucide` before "done".
3. **localStorage resize keys** `bodhi.${resizeKey}.{sideW,railW}` â€” stable per-section `resizeKey`;
   don't collide with `'bodhi-ui-theme'`.
4. **Chat last**: streaming/IndexedDB/abort/MCP-tools/self-scroll vs shell `overflow:hidden`
   `100dvh` grid + `is-fill` body â€” validate scroll fill before wiring the rail.
5. **Strip residual `data-theme` setattr** in ported pages (theme owned by `ThemeProvider`).
6. **Trailing-slash routing** (`trailingSlash:'always'`): new routes + all `<Link>`/nav hrefs
   trailing-slashed; regen `routeTree.gen.ts` when adding routes.
7. **`<Shell>` boundary**: setup/login/request-access/auth stay bare; verify `AppInitializer`
   redirects + the setup-status redirect test still pass after `__root` drops `AppHeader`.
8. **Flag cleanup is part of the batch**, not "later" â€” a section isn't done until its flags are
   removed and its old code deleted.

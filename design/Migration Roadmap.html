01 · The approach

Strangler, not big-bang

Your instinct — group by feature, then sequence — is right. I'd add one move in front of it: migrate the chrome before the content. Wrap the existing app in the new shell so old and new screens coexist under one navigation frame, then replace screen interiors as self-contained pull requests. Every step ships; nothing is a cliff.

03 · The sequence

Six phases, increasing stakes

Front-load the learning on low-blast-radius surfaces, climb in complexity, and finish on the highest-traffic surface only once the patterns are battle-tested. Each phase reuses what the one before proved.

Phase 0 · Foundation Coexistence layer no screens redesigned

Two reversible drops. Ship the design tokens (colors_and_type.css) app-wide — instant color/type modernization on screens that consume tokens, no layout risk. Then mount AppShell as the outer frame so existing routes render inside the new sidebar/header/nav. This is the keystone: after it, every screen migration is just swapping an inner panel.

04 · Why this order

The sequencing lens

Risk-adjusted, not alphabetical

Order features by risk × traffic. Cheap-to-get-wrong surfaces go first so the team learns the patterns where mistakes don't hurt; the surface users touch most goes last, on a proven base.

Each phase feeds the next

API Keys proves list + form. Settings adds banners + dialogs. Models proves the detail rail. MCP and Chat inherit all of it. No phase starts a pattern from scratch.

Migrate whole sections

Finishing a nav section before moving on avoids a half-redesigned area and lets you delete the old section's code in one piece — the cleanup the migration notes already call for.

Within a feature, safe screens first

Read-only and list screens before create forms before destructive/admin actions. Each ships behind its own flag — dogfood internally, then ramp.

05 · Hold the line

Cross-cutting guardrails

Lifted straight from your own migration notes — the things that bite when many screens share one foundation. These are the per-screen and global checks every phase runs.

DoDOne AppShell per screen. Renders a single shell, correct section/subPage highlight, no console errors, background-verified.
↔Shell behaviors must pass. Collapse-to-icon-rail, hover-reveal column resize, and responsive overlay drawers under 768px all work before a screen is called done.
⊞Never transition grid-template-columns with var() tracks — Chromium sticks the old width. Collapse is instant by design.
◆Tokens, not literals. Color/type/spacing come from the design tokens; page CSS holds only page-unique rules.
⚠Shared files are global. After touching the shell or a shared sheet, re-verify a sample of every screen type that consumes it — not just the one in front of you.

A recommendation, not a mandate — sequencing is easy to re-rank if business priorities pull a feature forward. The one piece I'd keep fixed: Phase 0 foundation first, Chat last. Open questions worth settling before Phase 1: production stack & router (does AppShell drop in as a layout route?), feature-flag tooling, and how aggressively we can dogfood each section internally.

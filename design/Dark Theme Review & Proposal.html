00 — SUMMARY

The short version

Dark mode looks broken not because any single page is wrong, but because the shared color tokens behind every page put the sidebar, header, main content, and cards inside a ~4% lightness band at the very bottom of the scale. At those luminance values the human eye can't separate them, so the whole app reads as one flat near-black void held together by hairline borders. A second, compounding problem: components hardcode light-mode hex values (dark pink/green/indigo text) that never flip, so secondary labels, status text, and the entire Tweaks panel silently fail in dark.

The fix in one breath

Widen the surface ramp so elevation reads as lightness (cards lighter than content, content lighter… no — content is the base; chrome and cards lift above it), bump secondary-text and border contrast into a comfortable-but-calm range, lightly desaturate accents for dark, and replace every hardcoded component color with a token. Six moves, mostly in colors_and_type.css and bodhi-app-shell.css.

1 · Widen the ramp

Five distinct elevation surfaces spanning 8%→21% lightness instead of four crammed into 5%→14%.

2 · Lift cards with light

Elevation = a lighter surface + a subtle border. Shadows do almost nothing on dark and are demoted to popovers only.

3 · Readable, calm text

Secondary text 68%→73%, borders 18%→25% lightness. Crosses AA without the harsh glare of pure white.

4 · Tune accents

Lotus pink lightly desaturated so it stops vibrating; accent washes become translucent so brand color survives on dark.

5 · Kill hardcoded hex

~40 component colors (dark pink/green/indigo text, light washes) replaced with theme-aware tokens.

6 · Fix the Tweaks panel

It's 100% light-hardcoded today. Rebuild on tokens so it's legible in both themes.

01 — DIAGNOSIS

Why the current dark theme fails

The architecture is good: there's a single token file (colors_and_type.css) and a single shell (bodhi-app-shell.css) that almost every page consumes through hsl(var(--…)). That's exactly why this is fixable centrally. The values inside those tokens are the problem.

Problem 1 — Every surface lives in the same 4% band

Here is where today's dark surfaces actually sit on the 0–30% lightness scale. Background, sidebar, header, and base cards are all crushed into the far-left corner:

Current dark surfaces — lightness (L%)
sidebar · bg · card · surface-2 all here → imperceptible
bg 6%
card / s2 9%
s3 14%
border 18%
  • Card is only +3% over background. At the bottom of the luminance curve a 3% delta is below the threshold most people can perceive — so cards don't look raised, panels don't look like panels.
  • --surface-2 equals --card (both 9%). Two named elevation levels resolve to one color, so the system can't express "slightly raised."
  • Sidebar is darker than the content it frames (the --sidebar token is 5%) — and worse, the shell doesn't even use it: .shell-sidebar is painted with --card, leaving the sidebar tokens dead. Either way, chrome and content never visually separate.
  • Borders are the only thing doing the work. Remove the 1px hairlines and the layout dissolves. That's the tell-tale sign of missing elevation.
What this looks like

In the Chat screenshot the left rail, the message column, and the right parameters rail are three different surfaces — but they read as one continuous black sheet. The "You" bubble and the tool-call card barely lift off the page. On the Users / Tokens / Settings pages the effect is even flatter because they have fewer colored accents to break it up.

Problem 2 — Components hardcode light-mode colors that never flip

Many components bypass the tokens and write raw hex chosen for light mode. These do not respond to the theme at all, so in dark they're either invisible or low-contrast. A non-exhaustive audit of the CSS/JSX found:

  • Dark-on-light text colors used as foreground — #B02A52 (dark lotus), #1F7D3F/#2a9b36 (dark green), #B4500A (dark saffron), #3E4AA8 (indigo). Found in active tabs, model tags, inline code, chips, "local/cost" meta, connected-status text — ~30+ occurrences across 8 files. All stay dark on a dark surface.
  • Light-wash backgrounds — e.g. the chat tool-call card uses rgba(238,240,251,.5) (a pale indigo wash) and a fixed #0B0D1A code block. Brittle: one is a light-mode wash on dark, the other a fixed near-black that ignores theme entirely.
  • The Tweaks panel is fully light-hardcoded — rgba(255,255,255,.6) field backgrounds, rgba(0,0,0,.1) borders, #fff thumbs. In dark it renders as glaring white controls.
  • Status greens have partial dark overrides on some pages and none on others — inconsistent because the rule lives per-page instead of in one token.

Problem 3 — Calibration & polish gaps

Saturated accents vibrate

The lotus primary is 347 100% 78% — full saturation. Bright saturated hues optically "shimmer" against dark surfaces and strain the eye; best practice is to desaturate for dark.

Secondary text is borderline

--muted-foreground at 68% sits right at the edge of comfortable on the darkest surfaces, and most secondary labels render on the darkest plane.

Shadows are doing nothing

The dark shadow tokens are tuned with black rgba — invisible against a near-black background. Depth has no fallback because elevation-by-lightness isn't set up.

No "highest" surface

Popovers, menus, and tooltips use --popover (10%) — barely above cards. Floating UI should clearly out-elevate the page beneath it.

02 — PRINCIPLES

What good dark themes do

These are the load-bearing rules from Material Design's dark theme guidance and the broader practitioner consensus. The proposal applies each one to Bodhi specifically.

P1

Dark grey, never pure black

Black can't show shadows or elevation and causes a harsh "halation" glow under light text. Use a dark, slightly-tinted grey base (Material's reference is #121212). Bodhi keeps a faint indigo tint — on-brand and endorsed for screens.

P2

Elevation = lighter surface

Shadows barely register on dark. Communicate hierarchy by making higher surfaces lighter. The higher the layer, the lighter it gets — a strict "lighter-on-top-of-darker" stack.

P3

Desaturate accents

Saturated colors vibrate and often fail AA on dark. Shift toward lighter, less-saturated tones (the 200–50 range of a palette) so brand color stays legible and calm.

P4

Don't use pure white text

#FFFFFF on dark shimmers. Use a slightly dimmed near-white, then step secondary/tertiary text down in opacity-equivalent lightness for a clean hierarchy.

P5

Enough contrast at every layer

Body text should clear WCAG AA (4.5:1) on the lightest surface it can land on. Material targets ≥15.8:1 of white-on-base so headroom survives as surfaces lift.

P6

Limited color, mostly surface

Most of a dark UI should be dark surface with sparse, intentional accent. Tint accent fills as translucency over the surface so they read at any elevation.

Sources: Material Design 2 — Dark theme; Nielsen/UX-Planet & Toptal dark-UI guides; practitioner write-ups on scalable token architectures. Summarized, not quoted.

03 — PROPOSED SYSTEM

The new dark foundation

A semantic elevation ramp replaces the flat surface set. You choose by role ("this is a card", "this is a popover"), not by guessing a lightness. Each step is a clearly perceptible jump, all warm-indigo-tinted, all low-chroma so they read as sophisticated slate-navy rather than saturated blue.

Surface elevation ladder

--background234 17% 8%App base canvas · main content well (deepest plane)
--surface-1233 16% 10%Sunken: inputs, code blocks, table zebra
--surface-2 / --card232 14% 13%Base raised: cards, sidebar, rail, header chrome
--surface-3231 13% 17%Hover, nested card, selected row
--surface-4 / --popover230 12% 21%Highest: menus, dropdowns, tooltips, modals
Why this works

Spread of 8%→21% gives five steps that are each individually distinguishable, instead of four steps inside 5%→14%. The main content stays the darkest plane (calm, low-glare); sidebar/header/cards lift above it; floating UI sits clearly highest. Borders become a supporting hairline, not the only separator.

Text & contrast — "balanced, readable, calm"

--foreground233 14% 95%Primary text — warm near-white, not #FFF (no halation)
--muted-foreground231 12% 73%Secondary labels — up from 68%, clears AA on base
--faint-foreground231 10% 55%Tertiary / placeholder / disabled (new token)

Borders & dividers

--border231 11% 25%Default separators — up from 18%, now visible without the band
--border-strong231 11% 32%Input edges, emphasized dividers (new token)

Accents — desaturated + translucent

You asked to keep the lotus identity but were open to tuning and to translucent fills. Both: the solid primary drops from 100%→88% saturation (stops the shimmer), and accent backgrounds become translucent overlays so brand color reads correctly at any elevation — and so the same token works in light and dark.

Lotus · primary

347 88% 78% solid for buttons. Fills: lotus 16% / surface. Text-on-dark uses the light #FFA4B8, never #B02A52.

Semantic states, all shifted lighter for dark. Status text uses light variants via one shared token.

Elevation, shadow & focus

  • Depth comes from the ramp + a hairline border, applied consistently. Shadows are kept only for genuinely floating layers (popovers, menus, modals, toasts) and paired with a subtle light top-edge so they register.
  • Focus rings switch to a lotus ring at higher opacity so keyboard focus is unmistakable on dark.
  • Optional: a 1px hsl(0 0% 100% / .04) inner top highlight on raised cards (the "overlay" trick) for extra tactility. Easy to toggle.

05 — FRAMEWORK CHANGES

What actually changes, and where

Most of the work is in two shared files. The rest is a mechanical sweep replacing hardcoded hex with tokens. Tags: FRAMEWORK shared tokens · LAYOUT shell/structure · COMPONENT per-page CSS.

colors_and_type.cssFW
Rewrite the .dark token block: new 5-step surface ramp, raised border/text contrast, desaturated accents, new --surface-4, --border-strong, --faint-foreground. The single highest-leverage change.
bodhi-app-shell.cssLAY
Map shell regions to the new ramp: main = --background, sidebar/rail/header = --card, popovers/menus/tooltips = --surface-4. Add dark elevation-overlay + focus-ring rules. Retire the dead --sidebar tokens (or wire them up).
— shared accent helpersFW
Consolidate the duplicated --c-lotus/saffron/leaf/indigo/teal blocks (copy-pasted into 5+ files) into one source of truth with correct dark *-text values, so status/chip text never hardcodes dark hex again.
tweaks-panel.jsxCMP
Rebuild its inline styles on tokens (fields, borders, thumbs, segmented control). Currently 100% light-hardcoded → unusable in dark.
bodhi-chat.cssCMP
Replace hardcoded tool-call wash, fixed code-block bg, and #B02A52 / #1F7D3F / #3E4AA8 text in tabs, model tags, inline code, meta.
bodhi-models.cssCMP
Chip text, status greens, key-ok text → tokens. Solid .act-use / .btn-add backgrounds already pink; keep, but use token.
access-request(s) · local-model · create-fallback · mcp-*CMP
Each carries a copy of the accent block with light-mode *-text values + a few #fff/dark-hex spots. Point them at the shared helpers; spot-fix remaining literals.
All pages (verification)LAY
Toggle each of the 13 screens in dark and confirm: chrome≠content, cards lift, all text ≥AA, no white controls, accents calm.
Backwards-compatible by design

Because pages already consume hsl(var(--card)), var(--border), etc., editing the token values updates them with zero page edits. The component sweep only touches the minority of spots that bypassed the tokens. Light mode is untouched.
